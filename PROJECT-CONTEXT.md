## 1. What it is
Multi-tenant B2B SaaS that runs **autonomous Claude agents over inbound messaging channels**
for service businesses (celebrants, driving instructors, photographers, event planners).
Captures leads from WhatsApp / SMS / Instagram / voice (Vapi) / embeddable web chat →
classifies intent → answers from knowledge base → books meetings → takes Stripe deposits →
scores leads, with a **human-approval gate** for low-confidence replies. On top sits a booking
system, CRM, analytics, agency/white-label, billing, referrals, and a super-admin console.

**Stack:** Next.js 15 (App Router) · Supabase (Postgres + pgvector + Auth + Storage) ·
TypeScript strict · Tailwind v4 · Vercel. **Counts:** 314 TS/TSX files · 104 API routes · 21 migrations.

## 2. Inbound message lifecycle (the spine)
```
Channel webhook / web-chat POST
  → Adapter.normalizeInbound() → NormalizedMessage (channel-agnostic)
  → findOrCreateContact()      (dedup by phone, then email)
  → findOrCreateConversation() (one per contact+channel via external_thread_id)
  → store inbound message
  → processInboundMessage() [ORCHESTRATOR]
      1. load profile AI-quality settings (confidence_threshold, require_approval, memory_depth, language)
      2. buildPriorContext() — last N conversations as memory
      3. runConcierge() → Claude classify_message tool → {intent, sentiment, urgency, routing, language, confidence}
      4. route: direct_response | knowledge (RAG) | action (tool loop ≤5) | human_handoff
      5. composeResponse() — opt-out check, AI disclaimer on first msg, 1000-char cap
      6. APPROVAL GATE: require_approval OR confidence < threshold → ai_approvals(pending), shouldSend=false
      7. logAIInteraction() → ai_interaction_logs (tokens, latency, tools)
  → shouldSend ? deliver via adapter : sits in /approvals queue
```
**Layering:** thin API routes (auth + Zod + Supabase) → `src/lib/*` domain services →
Supabase (RLS). Server Components fetch+prop-pass; Client Components subscribe to Realtime + hold Zustand state.

⚠️ **Biggest functional gap:** orchestrator is wired **only for the web-chat path**
(`/api/chat/message`). Twilio/Meta/Vapi webhooks normalize + persist but do **not** call
`processInboundMessage()` yet ("Phase 3 TODO"). Non-web inbound lands in DB unprocessed.

## 4. Data model
- **Tenant/identity:** `profiles` (1:1 auth.users — holds **every integration key** + AI-quality + branding/referral/slack/digest),
  `team_members`, `team_invitations`, `sub_accounts`.
- **Messaging:** `contacts`, `channels` (config = **unencrypted JSONB**), `conversations`, `messages`.
- **AI:** `knowledge_bases`, `knowledge_documents`, `knowledge_chunks` (pgvector 1536-dim, IVFFlat cosine),
  `ai_agent_configs`, `ai_interaction_logs`, `ai_approvals`.
- **Booking:** `services`, `availability_schedules`, `blocked_dates`, `bookings`, `booking_settings` (unique slug).
- **Billing/ops:** `subscriptions`, `analytics_events`, `webhook_logs`, `webhook_endpoints`,
  `api_keys` (SHA-256), `notifications`, `audit_logs`, `feature_flags`, `referral_credits`, `broadcasts`.
- **RLS:** every tenant table keyed `user_id = auth.uid()`. Team members reach owner data via a
  `team_members` subquery → app abstracts as `ownerId`. Super-admin via `is_super_admin()` SECURITY DEFINER.
  Public SELECT on booking tables for unauth pages. Service-role inserts for `webhook_logs`/`notifications`/`audit_logs`.
- **Functions/triggers:** `handle_new_user()` (auto-profile), `update_contact_on_message()` (cold→warm + last_interaction),
  `match_knowledge_chunks()` (RAG, threshold 0.7 top-5), usage counters (**not enforced**),
  `generate_referral_code()` trigger, `updated_at` touch.

## 5. Cross-cutting conventions
- **Auth/IDOR:** every route calls `getAPIContext()`/`getAuthContext()` (`src/lib/auth/get-user.ts`),
  then filters **every** query `.eq("user_id", ctx.ownerId)`. `ownerId` (not raw user.id) = load-bearing
  tenant isolation. RLS is second line.
- **Validation:** Zod schema-first (`src/lib/validators/*`), `parse()` before DB, `z.infer<>` for types.
- **Errors:** `AppError` (statusCode+code), safe client msg / full server log, `app/error.tsx` boundary.
- **BYO key:** `override || ANTHROPIC_API_KEY || throw`; per-user key in `profiles.anthropic_api_key`;
  SDK cached per key (**no invalidation** — gotcha).
- **Realtime:** `use-realtime.ts` wraps `postgres_changes`, always filtered `user_id=eq.`, feeds Zustand.
- **Rate limit:** Upstash sliding-window; **degrades to allow-all when env absent** (dev = no limit).
- **Side effects:** webhooks/Slack/auto-advance fire-and-forget `void (async()=>{})()` — never block/rollback.

## 7. Gotchas / risks / tech debt
**Prod blockers:**
- 🔴 **`SKIP_AUTH=true`** in `.env.local` → hardcoded `DEV_USER` (`00000000-…-0001`) with super-admin + RLS bypass (service-role client). Must remove before prod. (`src/lib/supabase/server.ts`, `middleware.ts`, `get-user.ts`.)
- 🔴 **Migration version collision** — `20260417/18/19_*` files use day-only versions; `supabase db push` collides on `schema_migrations` PK. Workaround: apply via `docker exec psql`. **Renumber to unique 14-digit timestamps before any prod push.** Not yet pushed to remote.
- 🔴 **Dead cloud Supabase** (`cujeuerrfermubhonfyf` = NXDOMAIN) → dev runs on local stack (OrbStack/Docker), ports +100 (API 54421 / DB 54422 / Studio 54423), `npm run dev` on **port 3004**.
- `NEXT_PUBLIC_APP_URL` still localhost (used in booking URLs + webhook callbacks).

**Already fixed (don't reintroduce):** RLS infinite recursion (42P17) — `20260415_roles_and_teams.sql` super-admin
policies self-selected `profiles` inside a policy on `profiles` → every query touching profiles 500'd (masked by SKIP_AUTH).
Fixed in `20260604000001_fix_profiles_rls_recursion.sql` via `is_super_admin()` SECURITY DEFINER.

**Security flags from reviewers:**
- ⚠️ **Web-chat IDOR/priv-esc** — `/api/chat/message` uses the admin (RLS-bypass) client and trusts `businessId` (=user_id) from the client request with no ownership check.
- ⚠️ `channels.config` stores API keys/tokens as **plaintext JSONB** (only the per-account Anthropic key in `profiles` is encrypted).
- ⚠️ Impersonation cookie `httpOnly=false` (XSS-readable). Widget iframe `lf:close` postMessage has no origin check. Public booking relies on slug as only access control, no per-slug rate limit. Referral apply not rate-limited / not strictly idempotent. No subscription-limit enforcement. GDPR delete incomplete (orphaned attachments; export not streamed → OOM risk).

**Correctness/scaling debt:**
- Orchestrator not wired to Twilio/Meta/Vapi webhooks (web chat only) — biggest functional gap.
- In-memory message queue (`channels/queue.ts`) is single-process — needs BullMQ/Redis/Trigger.dev to scale.
- **Knowledge RAG is a placeholder** — docs upload `status=pending`, never chunked/embedded; `runKnowledgeAgent` gets empty chunks; vector search not invoked.
- Approval message-insert-after-approve is non-transactional (best-effort rollback).
- Anthropic client cache never invalidates — changed DB key keeps old SDK instance until restart.
- Phone/email dedup not normalized (no libphonenumber; case-sensitive email).
- Booking double-book guarded app-level only (no DB unique constraint); slug uniqueness app-checked.
- Analytics aggregation in-memory from raw rows (fine <100k rows).
- Calendar / Stripe-payment-link / `search_knowledge` tools are stubs (Phase 5).
